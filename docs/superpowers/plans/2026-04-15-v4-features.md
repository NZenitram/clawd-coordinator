# V4 Feature Plan: Profiles, Templates, Schedules, Webhooks, Pools

**Goal:** Five features that strengthen the core value proposition — Claude doing work on remote machines, orchestrated from anywhere — with less friction and more automation.

## Effort Summary

| Feature | Effort | New Deps | Dependencies |
|---------|--------|----------|-------------|
| Agent profiles | ~4.5h | None | None |
| Task templates | ~3-4h | None | None |
| Agent pools | ~3-4h | None | None |
| Scheduled tasks | ~2-3 days | `croner` | None |
| Webhook triggers | ~3-4 days | None | None |
| **Total** | **~8-10 days** | | |

Profiles, templates, and pools are independent and quick. Schedules and webhooks are larger but also independent. All five can be done in parallel.

---

## 1. Agent Profiles (~4.5h)

`coord agent --profile ops` loads a stored config instead of typing all flags.

**Config schema** — `agentProfiles` in `~/.coord/config.json`:
```json
{
  "agentProfiles": {
    "ops": {
      "url": "wss://host:8080",
      "token": "...",
      "name": "ops-agent",
      "cwd": "/home/user",
      "allowedTools": ["Read", "Write", "Edit", "Bash"],
      "addDirs": ["/etc", "/var/log"],
      "permissionMode": "auto",
      "maxConcurrent": 2
    }
  }
}
```

**CLI changes** — `--profile <name>` on `coord agent`. CLI flags override profile values. `--url`, `--token`, `--name` become optional when profile provides them.

**Management** — `coord profiles list|show|create|delete`

**Files:** config.ts (interface), agent.ts (merge logic), profiles.ts (new), index.ts

---

## 2. Task Templates (~3-4h)

`coord run --template upgrade-package --vars "package=openclaw,version=2026.4.14"` loads a reusable prompt template.

**Format** — JSON files in `~/.coord/templates/<name>.json`:
```json
{
  "name": "upgrade-package",
  "prompt": "Upgrade {{package}} to version {{version}}. Run tests after.",
  "on": "ops-agent",
  "budget": "1.50",
  "variables": {
    "package": { "required": true },
    "version": { "required": true, "default": "latest" }
  }
}
```

**Substitution** — `{{varName}}` regex replace. No library needed.

**CLI** — `--template <name>` and `--vars "key=value,..."` on `coord run` and `coord fan-out`. Prompt argument optional when template provides it.

**Management** — `coord templates list|show|create|delete`

**Files:** templates.ts (new, shared/), templates.ts (new, cli/commands/), run.ts, fan-out.ts, index.ts

---

## 3. Agent Pools (~3-4h)

`coord run "test" --pool staging` dispatches to the least-loaded agent in the pool.

**Registration** — `--pool <name>` on `coord agent`. Stored in AgentInfo and register payload.

**Selection** — Least-loaded strategy (fewest currentTaskIds relative to maxConcurrent). No additional state needed.

**CLI** — `--pool <name>` as alternative to `--on <agent>` on run/fan-out. Exactly one required. Fan-out with `--pool` dispatches to ALL agents in the pool.

**Listing** — `coord pools` groups agents by pool, shows counts and capacity.

**Server** — dispatch-task resolves pool to agent via `registry.pickFromPool()`. New `dispatch-fan-out` command for pool fan-out.

**REST/MCP** — Pool-aware dispatch, new `/api/fan-out` endpoint, `fan_out_pool` MCP tool.

**Files:** messages.ts (pool field), registry.ts (getPoolAgents, pickFromPool), server.ts, run.ts, fan-out.ts, pools.ts (new), rest.ts, mcp/server.ts, agent.ts, daemon.ts, index.ts

## 4. Scheduled Tasks (~2-3 days)

`coord schedule create --cron "0 6 * * *" --on ops-agent --prompt "Check service health"`

**Storage** — `schedules` SQLite table (extends existing sql.js pattern). Fields: id, cron_expr, agent_name, prompt, paused, budget, permissions, org_id, timestamps, run_count.

**Scheduler** — `Scheduler` class in coordinator. 15-second tick interval checks `next_run_at <= now` for non-paused schedules. Dispatches via existing task/queue pipeline. Computes next run via `croner` library (3KB, zero deps, ESM).

**Restart recovery** — On startup, recompute next_run_at for all schedules. Missed runs skip to next future occurrence (configurable).

**CLI** — `coord schedule create|list|delete|pause|resume`

**REST** — Full CRUD at `/api/schedules`

**Dep:** `croner` (lightweight cron parser)

**Files:** scheduler.ts (new), schedule.ts (new CLI), sqlite-store.ts (schedules table), rest.ts, server.ts, serve.ts, index.ts

---

## 5. Webhook Triggers (~3-4 days)

`coord webhook create --name on-push --on dev-agent --prompt "Pull latest and run tests on {{payload.ref}}" --secret mysecret`

**Storage** — `webhooks` SQLite table. Fields: id, name (unique), agent_name, prompt_template, secret_hash, org_id, timestamps, trigger_count.

**HTTP handler** — `POST /hooks/:name` on existing HTTP server. Authenticates via HMAC-SHA256 (GitHub `X-Hub-Signature-256` pattern). Parses webhook JSON, renders prompt template with `{{payload.field}}` dot-path resolution. Dispatches via existing task/queue pipeline.

**Template engine** — `{{payload.ref}}`, `{{payload.commits[0].message}}` dot-path resolver. Regex + object walk, ~20 lines.

**Rate limiting** — Per-webhook token bucket (10 req/min default).

**CLI** — `coord webhook create|list|delete|test`

**REST** — CRUD at `/api/webhooks` + `POST /api/webhooks/:name/test` dry-run

**Files:** webhook-store.ts (new), webhook-template.ts (new), webhook.ts (new CLI), rest.ts, server.ts, index.ts

---

## Recommended Execution Order

```
Agent profiles (4.5h) ──────┐
Task templates (3-4h) ──────┤
Agent pools (3-4h) ─────────┼── All independent, can parallelize
Scheduled tasks (2-3 days) ─┤
Webhook triggers (3-4 days) ┘
```

Start with profiles + templates + pools (quick wins, immediate usability improvement). Then schedules and webhooks.
